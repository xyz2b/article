# x86 的分段：一个被 Linux 绕过、却移除不掉的历史包袱

最近一段时间写了好几篇，一路从"Vec 在内存哪"写到缺页中断的内核处理路径，反复出现一个词——"虚拟地址"。但我一直没提另一件事：在 x86 上，分页不是地址翻译的起点，它前面还压着一层更古老的机制：**分段**。

我们打印了那么多地址，从来没操心过"这个地址属于哪个段"。CPU 里明明有 6 个段寄存器（CS/DS/SS/ES/FS/GS），现在到底还装着什么？这篇就把它们找出来，直接打印，让 CPU 自己告诉我们分段的死活。

> 这次的环境和前几篇不一样。前面跑的是本机 macOS / arm64，但**分段是 x86 独有的东西**，arm64 上压根没有这套段寄存器。所以这篇所有数字都是在一个 x86-64 的 Linux 容器里真跑的（docker `gcc:13`，`--platform linux/amd64`）。复制代码、在任意 x86-64 Linux 上 `gcc segdemo.c && ./segdemo` 即可复现。

下面这段程序，把 6 个段寄存器的值原样读出来，再把还活着的那几个拆开看：

```c
#define _GNU_SOURCE
#include <stdio.h>
#include <asm/prctl.h>
#include <sys/syscall.h>
#include <unistd.h>

// 一个线程局部变量：编译器会用 fs 段来给它寻址（后面第四节会用到）
__thread int tls_var = 12345;

// 把一个段选择子（段寄存器里那 16 位）按 x86 规定的格式拆开：
//   bit 3 及以上 = index（在描述符表里的第几项）
//   bit 2        = TI （0=查 GDT，1=查 LDT）
//   bit 0~1      = RPL（请求/当前特权级：0=内核态，3=用户态）
static void decode(const char *name, unsigned long sel){
    printf("  %s = 0x%02lx   -> index=%lu  TI=%lu(%s)  RPL=%lu(%s)\n",
        name, sel, sel >> 3, (sel >> 2) & 1,
        ((sel >> 2) & 1) ? "LDT" : "GDT",
        sel & 3, (sel & 3) == 3 ? "用户态" : (sel & 3) == 0 ? "内核态" : "?");
}

int main(void){
    unsigned long cs, ds, ss, es, fs, gs;
    asm("mov %%cs,%0" : "=r"(cs));
    asm("mov %%ds,%0" : "=r"(ds));
    asm("mov %%ss,%0" : "=r"(ss));
    asm("mov %%es,%0" : "=r"(es));
    asm("mov %%fs,%0" : "=r"(fs));
    asm("mov %%gs,%0" : "=r"(gs));

    printf("== 六个段寄存器现在装着什么 ==\n");
    printf("  CS=0x%02lx DS=0x%02lx SS=0x%02lx ES=0x%02lx FS=0x%02lx GS=0x%02lx\n",
           cs, ds, ss, es, fs, gs);

    printf("\n== 还活着的两个：拆开看 ==\n");
    decode("CS", cs);
    decode("SS", ss);

    printf("\n== fs/gs 的选择子废了，但 base 还在当线程指针 ==\n");
    unsigned long fsbase = 0, gsbase = 0;
    syscall(SYS_arch_prctl, ARCH_GET_FS, &fsbase);
    syscall(SYS_arch_prctl, ARCH_GET_GS, &gsbase);
    printf("  fs 选择子 = 0x%lx   FSBASE = 0x%lx\n", fs, fsbase);
    printf("  gs 选择子 = 0x%lx   GSBASE = 0x%lx\n", gs, gsbase);
    printf("  &tls_var  = %p   <- 落在 FSBASE 旁边\n", (void*)&tls_var);
    return 0;
}
```

```bash
gcc -O0 segdemo.c -o segdemo && ./segdemo
```

跑出来是这样：

```
== 六个段寄存器现在装着什么 ==
  CS=0x33 DS=0x00 SS=0x2b ES=0x00 FS=0x00 GS=0x00

== 还活着的两个：拆开看 ==
  CS = 0x33   -> index=6  TI=0(GDT)  RPL=3(用户态)
  SS = 0x2b   -> index=5  TI=0(GDT)  RPL=3(用户态)

== fs/gs 的选择子废了，但 base 还在当线程指针 ==
  fs 选择子 = 0x0   FSBASE = 0x7fffff7c2640
  gs 选择子 = 0x0   GSBASE = 0x0
  &tls_var  = 0x7fffff7c263c   <- 落在 FSBASE 旁边
```

第一行就够耐人寻味了：**DS、ES、FS、GS 全是 0。** 数据段寄存器全空着。可程序明明在不停地读写数据，没出任何问题。而 CS、SS 还有值（0x33、0x2b）。

为什么有的清零、有的留着？要讲清这件事，得先回到分段当年是干嘛的。

## 一、分段的由来：寄存器装不下地址的年代

分段不是为内存保护发明的，最早纯粹是被逼出来的一个寻址补丁。

回到 8086（1978 年）。它是 16 位的 CPU，寄存器只有 16 位宽。16 位能表示的地址范围是 `2^16 = 64KB`——可这块芯片的地址总线有 20 根，物理上能接 `2^20 = 1MB` 内存。

矛盾就在这：**寄存器只数得到 64KB，内存却有 1MB。** 一个 16 位寄存器，根本装不下一个能指向整个 1MB 空间的地址。

Intel 的解法是「段 : 偏移」：再拿一个 16 位寄存器当**段基址**，把它左移 4 位（乘 16），再加上 16 位的**偏移**，拼出 20 位的物理地址：

```
   8086 实模式：两个 16 位寄存器拼出 20 位物理地址

   段寄存器  0x1234  ──左移4位(×16)──►  0x12340
                                          +
   偏移      0x5678  ──────────────────►   0x5678
                                       ─────────────
   物理地址                              0x179B8   (20 位，够到 1MB)
```

段寄存器就是这么来的：它是为了让 16 位的小寄存器够得着 1MB 内存而硬塞进来的一个「高位补丁」。CS 管代码取指用哪段、DS 管数据读写用哪段、SS 管栈用哪段、ES 是备用数据段。每次访问内存，CPU 都自动把对应的段寄存器拿来左移相加。

就像我们用"xx省xx市xx路xx号"拼出一个地址——省市路是高位，门牌号是偏移，两段数拼起来才够定位一个地方。8086 的段:偏移做的是完全一样的算术，只不过当时没有比这更好的办法。

所以"段"最早不是什么设计哲学，就是个**位数不够、拿另一个寄存器凑高位**的土补丁。和保护、权限毫无关系。

## 二、保护模式：段不再是左移，而是查一张表

到了 80286、386，事情变复杂了。CPU 进入**保护模式**，地址变宽（32 位能直接寻址 4GB），「段左移 4 位」这个补丁不再需要了。但 Intel 没扔掉段寄存器，而是给它换了个全新的、严肃得多的身份。

保护模式下，段寄存器里装的不再是"基址的高位"，而是一个**选择子（selector）**——一个编号，指向一张表里的某一项。这张表叫 **GDT**（Global Descriptor Table，全局描述符表），由操作系统建好、告诉 CPU 它在哪。表里每一项叫一个**段描述符**，描述一个段长什么样。

选择子怎么指向表项？就是开头代码里 `decode` 拆的那三段：

```
   段选择子（段寄存器里的 16 位），以 CS=0x33 为例：

   0x33 = 二进制 0000000000110 0 11
                 └──index=6──┘ │ └RPL=3
                              TI=0
   bit 3 及以上：index = 6   → 查 GDT 的第 6 项
   bit 2       ：TI    = 0   → 查 GDT（=1 则查另一张表 LDT）
   bit 0~1     ：RPL   = 3   → 请求以特权级 3（用户态）访问
```

CPU 拿着 index 去 GDT 里取出第 6 项那个段描述符。**这一篇你只需要知道描述符里有三样关键的东西**（它一共 64 位，逐位结构留到后面专门讲）：

- **base（段基址）**：这个段从哪个地址开始。
- **limit（段界限）**：这个段有多大，越界就报错。
- **DPL（Descriptor Privilege Level，描述符特权级）**：访问这个段需要的特权级别（0 = 内核，3 = 用户）。

于是保护模式下访问内存变成了：拿段寄存器里的选择子查 GDT，取出描述符，用里面的 base 当起点、加上偏移得到线性地址，同时拿 limit 查越界、拿 DPL 查权限。**段终于有了"边界"和"权限"的含义**——这才是它名字里"保护"的由来。

```
   保护模式：段寄存器 → 查表 → 拿 base/limit/DPL

   DS 选择子 ──index──► GDT 第 N 项（段描述符）
                          ├─ base   = ?   段从哪开始
                          ├─ limit  = ?   段多大（越界报错）
                          └─ DPL    = ?   要什么权限才能碰
                                   │
              地址 = base + 偏移 ◄─┘   （再交给分页，翻译成物理地址）
```

注意最后那一步：分段算出来的结果（base + 偏移）叫**线性地址**，它还要再经过**分页**才变成真正的物理地址。所以在 x86 上，地址翻译本来是**两道关**：先分段，再分页。我们前面几篇讲的全是第二道关。这篇要回答的，是第一道关后来怎么了。

## 三、Linux 的选择：把所有段的 base 设成 0

现在回头看那个 `DS=0x00`。

到了 x86-64（长模式），Intel 自己先动了刀：**CPU 在 64 位模式下，干脆忽略 CS/DS/SS/ES 这几个段描述符里的 base 和 limit，一律当 base=0、limit=无限大处理。** 也就是说，硬件层面分段的"算术"部分已经被跳过了——不管段寄存器指向哪，数据访问的段基址都是 0。

而 Linux 的做法和这个方向完全一致，甚至在 32 位时代就这么干了：**它建的 GDT 里，几个常用段的 base 全设成 0，limit 拉满整个地址空间。**

base 都是 0 意味着什么？意味着「段 : 偏移」这步退化成了：

```
   平坦模型：所有段 base = 0

   线性地址 = base + 偏移
            =   0   + 偏移
            =        偏移            ← 段这一步什么也没做

   你写下的地址，原样就是线性地址，再直接进分页。
```

**段基址是 0，加了等于没加。** 分段这道关，被 Linux 拉成了一条直线——所有段从同一个 0 起点出发，覆盖同一整片地址空间，彼此完全重叠。这就是**平坦内存模型（flat memory model）**：在程序眼里根本没有"段"这回事，整个内存就是一根从 0 开始的笔直地址轴。我们前面几篇画地址空间时画的那根直轴，能成立的前提就在这里。

这也解释了 `DS=ES=0`：既然所有数据段都 base=0、行为完全一样，Linux 索性懒得给 DS、ES、FS、GS 设有意义的选择子，直接塞 0。数据访问压根不看它们了。

想象你家门牌原来是"3栋205室"，楼号 3 是段基址，205 是偏移。现在所有楼统一改叫"0栋"，那"0栋205室"退化成什么？就是 205 室。**楼号加了等于没加。**

Linux 对所有段做的正是这件事——把所有 base 填成 0。

这就是分段"消失"的真相：它没有被从 CPU 里移除，而是被 Linux 用一个**「所有 base 都填 0」**的动作，整个设成了空操作。硬件那套 GDT、段描述符、段寄存器全都还在、CPU 每次访存还在老老实实走分段这道关——只不过这道关被设成了"什么都不做"，你穿过它毫无感觉。我们前几篇能心安理得地把地址当成一根直轴来画，正是因为有人在底下把段全压成了 0。

## 四、两处还在真正干活

如果分段被设成这么彻底的空操作，那 CS、SS 为什么还留着值（0x33、0x2b）？FSBASE 为什么非零？

因为分段这套机制里，有两样东西 Linux **还在用**，删不掉。

**第一处：特权级。** 这是段寄存器现在最重要的活。

回头看 CS 拆出来的 `RPL=3`。这 2 个 bit 就是 CPU 此刻所处的特权级别——3 是用户态。当 CPU 陷入内核（比如你发起一次系统调用），内核代码段的选择子是 `__KERNEL_CS=0x10`，拆开看 RPL=0，也就是内核态。

```
   特权级活在 CS/SS 选择子的低 2 位

   用户态运行你的代码：  CS=0x33  低2位=3  ── ring 3（用户态）
                        SS=0x2b  低2位=3
   陷入内核执行系统调用：CS=0x10  低2位=0  ── ring 0（内核态）
                        SS=0x18  低2位=0

   CPU 拿"当前特权级"和要访问的段/页的 DPL 一比：
     用户态(3) 想碰只有内核(0) 能碰的东西 → 直接拒绝、触发异常
```

这就是 ring 0 / ring 3 的物理实现：CPU 时刻知道自己跑在哪个特权级（就看 CS 低 2 位），每次访问受保护的资源，都拿这个级别和资源要求的 DPL 比一下，不够格就拦下来。**用户态程序碰不到内核内存、不能直接执行特权指令，根子就在这 2 个 bit 上。** 分段整套机制里，这是 Linux 一天都离不开的部分。

> CPU 里有一张通行证，通行证上有个数字——0 是最高权限（内核），3 是普通用户。这个数字就藏在 CS 最末尾的两个格子里。CPU 每次想进一扇门，先看这两个格子，数字不够就挡回去。

> 至于用户态怎么"合法地"切到内核态（`syscall` 指令一执行，CPU 怎么把 CS 从 0x33 换成 0x10、怎么换栈），那是另一套配合（syscall/sysret、中断门、TSS）。这里先按住，留给后面讲系统调用时专门拆。这篇只需要记住：**特权级是分段留给现代系统的头号遗产。**

**第二处：FS/GS 的 base。** 这是个更妙的"废物利用"。

看那组 fs/gs 的数据：`fs 选择子 = 0x0`，选择子是 0、看着像彻底废了；可 `FSBASE = 0x7fffff7c2640`，非零。而那个线程局部变量 `&tls_var = 0x7fffff7c263c`，正好就**贴在 FSBASE 旁边**（差了几个字节）。

```
   FS 的选择子废了，但它的 base 寄存器被征用了

   fs 选择子 = 0x0           ← 当段选择子用？废了
   FSBASE   = 0x7fffff7c2640 ← 但 base 本身非零，指向本线程的私有数据区
   &tls_var = 0x7fffff7c263c ← __thread 变量就摆在这个 base 旁边

   访问 tls_var ≈ 访问 [FSBASE + 固定偏移]
```

x86-64 在这里做了个取舍：它抛弃了 FS/GS 当普通段选择子的用法（所以选择子是 0），但**单独保留了它们的 base 寄存器**，让你能用一条 MSR 指令直接给 FSBASE/GSBASE 写一个任意地址。Linux 拿它干什么用？**线程局部存储（TLS）。**

每个线程有自己的一份 `__thread` 变量。Linux 让每个线程的 FSBASE 指向它自己的私有数据区，于是编译器访问 `tls_var` 时，生成的就是「FS 段基址 + 固定偏移」这种寻址——同一条指令、同一个偏移，在不同线程里因为 FSBASE 不同，自动落到各自的那份变量上。这正是 `&tls_var` 紧贴 FSBASE 的原因。（内核那侧则常用 GS base 指向每个 CPU 的私有数据。）

所以 FS/GS 是分段还在干活的第二个证据：**段选择子那套被丢了，但 base 寄存器被改造成了「线程私有数据的指针」**，一个本来要淘汰的零件，换了个岗位继续上班。

> 每个线程有一张便利贴，便利贴贴在哪里，哪里就是这个线程的私有数据。FSBASE 就是这张便利贴。FS 寄存器原本是用来查段表、找段在哪的（和 CS/DS 一个路子），现在这条路被废了——但便利贴还贴着，私有数据照样找得到。

把这篇收一下，其实是一条很清晰的演变线：

- **分段的出身很将就**：8086 上 16 位寄存器够不着 1MB 内存，拿一个寄存器左移凑高位，仅此而已，和保护毫无关系。
- **保护模式给了它身份**：段寄存器变成查 GDT 的选择子，描述符里的 base/limit/DPL 让段第一次有了边界和权限。
- **x86-64 + Linux 把它绕过去了**：所有段 base=0、平坦模型，分段这道关被设成"什么都不做"。我们前几篇能把整个内存当成一根从 0 开始的直轴、从来不管"段"，根就在这——有人在底下把段全压成了 0。
- **但两处还在真正干活**：特权级（CS/SS 低 2 位的 ring 0/3）是 Linux 天天在用的头号遗产；FS/GS 的 base 被改造成线程私有数据（TLS）的指针。CPU 出于向后兼容也一直保留着它。

所以 `DS=0` 和 `CS=0x33` 这两个值，正好各自讲了半个故事：**`DS=0` 是分段被绕过的那一半，`CS=0x33` 是它还在干活的那一半。** 一个寄存器清零，一个带着 ring 3 的印记继续跑——分段没有被移除，只是收缩到两个真实用途，其余用"填 0"跳过。

> 这篇我们说分段算完，把结果（线性地址）交给了**分页**。可分页这道关，在 x86 硬件上到底长什么样？CPU 靠哪个寄存器找到页表（CR3）、那棵 4 级页表树在硬件眼里怎么走、页表项那 64 位里每一位（present、读写、用户/内核、NX……）各是什么——这是这个系列下一篇要拆的：**分页硬件**。我们前面在 Linux 那条线上把分页的"行为"讲透了，这回换个视角，看 x86 把它"刻"成了什么样的电路约定。
