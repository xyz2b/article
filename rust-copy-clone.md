# Copy 比 Clone 便宜，为什么 Rust 还要求「先实现 Clone」？

几乎每个写 Rust 的人都敲过这行代码：

```rust
#[derive(Copy, Clone)]
struct Point { x: i32, y: i32 }
```

`Copy` 和 `Clone` 总是成对出现。时间久了你会发现一个规矩：想让一个类型支持 `Copy`，就必须同时实现 `Clone`，顺序还不能反过来。标准库把这层关系写死在了类型签名里：

```rust
pub trait Copy: Clone { }
```

第一次认真盯着这行看的人，十有八九会冒出一个困惑：`Copy` 是廉价的栈上按位拷贝，`Clone` 才是那个可能要在堆上重新分配内存、开销大得多的操作。既然 `Copy` 更便宜，为什么要让它依赖那个「更贵」的 `Clone`？这不是让简单的东西去背负复杂的前提吗？

这个困惑特别有代表性，因为它精准地踩在了一个认知错位上。想通它，你不只是搞懂了一条语法规则，而是会顺手摸清 Rust 整个 trait 体系的一条主线。

## 错位在哪：代价 vs 能力

困惑的根源是把两个不同维度的东西叠在了一起：**运行时代价**，和**类型能力**。

「Clone 比 Copy 贵」说的是运行时代价 —— 程序跑起来，`String::clone` 要去堆上申请一块新内存、把字节挨个拷过去，这确实比拷贝一个 `i32` 费事得多。

而 `Copy: Clone` 这行约束，说的根本不是代价，是能力。它表达的是一句纯粹的逻辑陈述：

> 任何能被 `Copy` 的类型，逻辑上一定也能被 `Clone`。

这里没有「先付出昂贵代价、再享受廉价操作」的意思。trait bound 描述的是「这个类型具备什么能力」，跟这些能力执行起来快还是慢，是两套完全独立的话。把它读成「便宜的依赖贵的」，就像看到「正方形是一种矩形」却抱怨「正方形明明更规整，为什么要依赖更宽松的矩形」一样 —— 你把分类关系当成了成本关系。

## Copy 是 Clone 的更强特例

那两者真正的关系是什么？是包含。

`Clone` 描述的能力是：**这个类型能复制出一份自己**。如何复制不限，你可以写任意逻辑，该分配堆内存就分配。

`Copy` 描述的能力则更苛刻：**这个类型不仅能复制，而且复制方式简单到编译器可以隐式地、按位地替你完成**，完成后原值还照样能用。

后者显然是前者的一个子集。一个类型如果连「能复制出一份」这个基本门槛都迈不过去，它绝无可能满足「能被隐式按位复制」这个更高的门槛。Copy 是 Clone 的加强版，加强版成立的前提是基础版先成立 —— 这是逻辑上的必然，不是 Rust 额外塞给你的负担。

`trait Copy: Clone` 这行，翻译成人话就是：要拿到「能隐式廉价复制」这张高级凭证，你得先持有「能复制」这张基础凭证。冒号在这里和泛型约束 `T: Clone` 里的冒号是同一个东西，都在表达「左边蕴含右边」。

## 对 Copy 类型来说，clone 根本不贵

困惑里还藏着一个隐含的错误前提：以为给 Copy 类型实现 Clone，会真的拖上「Clone 那份昂贵开销」。

不会。一个 `Copy` 类型的 `Clone` 实现，本质上就是按位拷贝它自己：

```rust
impl Clone for Point {
    fn clone(&self) -> Self {
        *self   // Copy 类型，解引用即按位复制，和隐式 copy 一模一样
    }
}
```

`point.clone()` 和编译器隐式做的那次 copy，生成的机器码是相同的，一样快。「Clone 很贵」这件事，只在 `String`、`Vec<T>`、`Box<T>` 这类持有堆资源的**非 Copy** 类型上才成立 —— 而它们本来就不允许实现 `Copy`。

所以「先实现 Clone」这个前提，对 Copy 类型来说没有引入一丝一毫的额外运行时成本。你担心的那个代价，从来没出现在这个场景里。

## 如果 Copy 可以独立存在，会塌掉什么

反过来设想一下：假如 Rust 允许一个类型只实现 `Copy`、不实现 `Clone`，会发生什么？

你会得到一个荒诞的局面 —— 存在一种类型，它能被隐式复制，却没法显式调用 `.clone()`。而 Rust 的泛型代码里，到处是这样写约束的：

```rust
fn duplicate<T: Clone>(x: &T) -> T {
    x.clone()
}
```

这个函数要求传进来的类型「能复制」。如果 `Copy` 不蕴含 `Clone`，那么一个最容易复制的 Copy 类型，反而无法满足 `T: Clone`、传不进这个函数 —— 一个「复制成本最低」的类型，被「要求能复制」的接口拒之门外。这显然说不通。

让 `Copy: Clone` 成立之后，所有 Copy 类型自动满足 `T: Clone`，泛型世界保持统一：凡是能 Copy 的，在任何要求 Clone 的地方都能用。约束的存在不是为了刁难，恰恰是为了让类型系统自洽。

## 这是一条贯穿标准库的主线

把这层「supertrait 表达能力包含」想明白，你会发现它在标准库里反复出现，规律完全一致：

```rust
trait Eq: PartialEq { }          // 完全相等是部分相等的更强特例
trait Ord: PartialOrd + Eq { }   // 全序是偏序的更强特例
trait Copy: Clone { }            // 隐式廉价复制是「能复制」的更强特例
```

每一对的结构都一样：**冒号右边是更宽松的能力，左边是更严格的特例，实现左边的前提是先满足右边。** 严格的那个，永远站在宽松的那个肩膀上。

## 一句话收尾

`Copy: Clone` 不是「做便宜的事得先付贵的代价」，而是「一种能力是另一种能力的特例，拥有前者必然拥有后者」。代价的高低，是 `String` 那类型才有的运行时现象，跟 Copy 类型无关 —— Copy 类型的 clone，和 copy 一样快。

记住这组对照就够了：

> **Clone 是「能复制」。Copy 是「能复制，且复制廉价到可以隐式发生」。后者天然包含前者。**
