# 一个线程被唤醒后，为什么不一定立刻运行——从 futex_wake 到 Linux 调度器

---

上一篇《[一个 pthread_mutex_lock() 到底锁了什么——从用户态 CAS 到内核调度](https://github.com/xyz2b/article/blob/main/CodeAdventure/一个pthread_mutex_lock到底锁了什么.md)》最后把 `pthread_mutex_lock` 的慢路径追到了这里：

```text
线程抢锁失败
→ futex(FUTEX_WAIT)
→ 线程睡眠
→ 持锁线程 unlock
→ futex(FUTEX_WAKE)
→ 线程被唤醒
→ 等调度器选中
→ 重新回到用户态抢锁
```

这里藏着一个容易误解的点：

> `futex_wake` 把线程唤醒了，不等于这个线程立刻开始运行。

那"唤醒"到底是什么意思？线程被唤醒后去了哪里？为什么明明锁已经放了，它还可能等几微秒甚至几毫秒才继续执行？

这一篇继续往下挖：**从 `futex_wake` 追到 Linux 的线程状态、运行队列、调度延迟和上下文切换。**

> 实验环境：Docker Desktop Linux 容器，`gcc:13` 镜像；内核 `Linux 6.12.65-linuxkit aarch64`，glibc `2.36-9+deb12u14`，GCC `13.4.0`。文中代码与输出都是真编真跑。具体 TID、纳秒数、上下文切换次数会随机器负载波动，结论看关系，不看绝对数字。这个内核没有暴露 `/proc/<tid>/sched` 的详细字段，所以本文用 `/proc/<tid>/schedstat`、`/proc/<tid>/status`、`/proc/<tid>/wchan` 和 `getrusage()` 做证据。

## 一、Linux 调度的不是 pthread_t，而是 task

用户态里我们说"pthread 线程"，内核里真正被调度的是一个个 `task_struct`。同一个进程里的多个 pthread，共享一个进程地址空间，但在 `/proc/<pid>/task/` 下会有各自的 TID。

先用代码证明。

<a id="code-tasks"></a>

```c
// 01_tasks.c
#define _GNU_SOURCE
#include <dirent.h>
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/syscall.h>
#include <unistd.h>

static pthread_barrier_t barrier;

static long gettid_linux(void) {
    return syscall(SYS_gettid);
}

static void *worker(void *arg) {
    long idx = (long)arg;
    printf("worker%ld: getpid()=%d  gettid()=%ld\n", idx, getpid(), gettid_linux());
    pthread_barrier_wait(&barrier);
    pthread_barrier_wait(&barrier);
    return NULL;
}

static int cmpstr(const void *a, const void *b) {
    const char *sa = *(const char * const *)a;
    const char *sb = *(const char * const *)b;
    return atoi(sa) - atoi(sb);
}

int main(void) {
    enum { N = 3 };
    pthread_t th[N];

    pthread_barrier_init(&barrier, NULL, N + 1);
    printf("main   : getpid()=%d  gettid()=%ld\n", getpid(), gettid_linux());

    for (long i = 0; i < N; i++) {
        pthread_create(&th[i], NULL, worker, (void *)i);
    }

    pthread_barrier_wait(&barrier);

    DIR *d = opendir("/proc/self/task");
    if (!d) {
        perror("opendir");
        exit(1);
    }

    char *tids[64];
    int n = 0;
    struct dirent *de;
    while ((de = readdir(d)) != NULL) {
        if (de->d_name[0] >= '0' && de->d_name[0] <= '9') {
            tids[n++] = de->d_name;
        }
    }
    qsort(tids, n, sizeof(tids[0]), cmpstr);

    printf("/proc/self/task:");
    for (int i = 0; i < n; i++) {
        printf(" %s", tids[i]);
    }
    printf("\n");
    closedir(d);

    pthread_barrier_wait(&barrier);
    for (int i = 0; i < N; i++) {
        pthread_join(th[i], NULL);
    }
    return 0;
}
```

编译运行：

```bash
gcc -O0 -pthread -o 01_tasks 01_tasks.c
./01_tasks
```

真实输出：

```text
main   : getpid()=29  gettid()=29
worker0: getpid()=29  gettid()=30
worker1: getpid()=29  gettid()=31
worker2: getpid()=29  gettid()=32
/proc/self/task: 29 30 31 32
```

四个线程的 `getpid()` 都是 `29`，说明它们属于同一个进程；但 `gettid()` 分别是 `29、30、31、32`，并且 `/proc/self/task` 下正好有这四个目录。

所以对 Linux 调度器来说：

```text
main thread  → task_struct(TID=29)
worker0      → task_struct(TID=30)
worker1      → task_struct(TID=31)
worker2      → task_struct(TID=32)
```

调度器挑的是这些 task，不是 C 语言里的 `pthread_t` 变量。

## 二、睡眠线程在哪里：status 和 wchan 能直接看到

上一篇说过：抢不到 mutex 的线程会进 `futex_wait` 睡眠。那睡眠状态到底能不能从内核接口里看见？

写一个程序：

- 主线程先持有 mutex。
- `waiter` 线程尝试 lock，抢不到，卡在 futex。
- `sleeper` 线程调用 `sleep(3)`，卡在定时器睡眠。
- 主线程读取两个线程的 `/proc/self/task/<tid>/status` 和 `wchan`。

<a id="code-states"></a>

```c
// 02_states.c
#define _GNU_SOURCE
#include <errno.h>
#include <pthread.h>
#include <stdatomic.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/syscall.h>
#include <time.h>
#include <unistd.h>

static pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
static pthread_barrier_t barrier;
static atomic_long waiter_tid;
static atomic_long sleeper_tid;

static long gettid_linux(void) {
    return syscall(SYS_gettid);
}

static void *waiter(void *arg) {
    atomic_store(&waiter_tid, gettid_linux());
    pthread_barrier_wait(&barrier);
    pthread_mutex_lock(&m);
    pthread_mutex_unlock(&m);
    return NULL;
}

static void *sleeper(void *arg) {
    atomic_store(&sleeper_tid, gettid_linux());
    pthread_barrier_wait(&barrier);
    sleep(3);
    return NULL;
}

static void print_field(long tid, const char *name) {
    char path[128];
    snprintf(path, sizeof(path), "/proc/self/task/%ld/status", tid);
    FILE *f = fopen(path, "r");
    if (!f) {
        perror(path);
        exit(1);
    }

    char line[256];
    while (fgets(line, sizeof(line), f)) {
        if (strncmp(line, "Name:", 5) == 0 || strncmp(line, "State:", 6) == 0 ||
            strncmp(line, "Tgid:", 5) == 0 || strncmp(line, "Pid:", 4) == 0) {
            printf("%s %s", name, line);
        }
    }
    fclose(f);

    snprintf(path, sizeof(path), "/proc/self/task/%ld/wchan", tid);
    f = fopen(path, "r");
    if (!f) {
        perror(path);
        exit(1);
    }
    if (fgets(line, sizeof(line), f)) {
        line[strcspn(line, "\n")] = 0;
        printf("%s wchan:\t%s\n", name, line);
    }
    fclose(f);
}

int main(void) {
    pthread_t twait, tsleep;
    pthread_barrier_init(&barrier, NULL, 3);

    pthread_mutex_lock(&m);
    pthread_create(&twait, NULL, waiter, NULL);
    pthread_create(&tsleep, NULL, sleeper, NULL);
    pthread_barrier_wait(&barrier);

    usleep(200000);
    printf("main tid=%ld\n\n", gettid_linux());
    print_field(atomic_load(&waiter_tid), "waiter ");
    printf("\n");
    print_field(atomic_load(&sleeper_tid), "sleeper");

    pthread_mutex_unlock(&m);
    pthread_join(twait, NULL);
    pthread_join(tsleep, NULL);
    return 0;
}
```

编译运行：

```bash
gcc -O0 -pthread -o 02_states 02_states.c
./02_states
```

真实输出：

```text
main tid=33

waiter  Name:	02_states
waiter  State:	S (sleeping)
waiter  Tgid:	33
waiter  Pid:	34
waiter  wchan:	futex_wait_queue

sleeper Name:	02_states
sleeper State:	S (sleeping)
sleeper Tgid:	33
sleeper Pid:	35
sleeper wchan:	hrtimer_nanosleep
```

两个线程的 `State` 都是 `S (sleeping)`，但 `wchan` 不同：

- `waiter` 卡在 `futex_wait_queue`，说明它确实睡在 futex 等待路径里。
- `sleeper` 卡在 `hrtimer_nanosleep`，说明它是普通的定时器睡眠。

这说明一件事：线程"睡眠"不是抽象说法。内核真的把这个 task 从运行队列里拿掉，挂到了某个等待点上。

对 mutex 慢路径来说，这个等待点就是 futex 等待队列：

```text
pthread_mutex_lock
→ CAS 失败
→ futex(FUTEX_WAIT)
→ task state = S
→ wchan = futex_wait_queue
```

## 三、TASK_RUNNING 不是"正在运行"

接下来是本文最关键的一点。

Linux 里 `TASK_RUNNING` 这个名字很容易误导人。它不是"这个线程正在 CPU 上执行"，而是：

> 这个 task 可以运行。

它包括两种情况：

```text
1. 正在某个 CPU 上运行
2. 在某个 CPU 的 runqueue 里排队，等调度器选中
```

所以 `futex_wake` 做的核心事情不是"让线程立刻继续执行"，而是把它从睡眠状态改成可运行状态，再放回运行队列。

可以把状态变化简化成这样：

```text
futex_wait queue
      │
      │ futex_wake
      ▼
TASK_RUNNING
      │
      ▼
CPU runqueue
      │
      │ scheduler pick
      ▼
真正上 CPU 执行
```

中间那段"在 runqueue 里排队"的时间，就是调度延迟。

Linux 给了一个很好用的观测接口：`/proc/<pid>/task/<tid>/schedstat`。它有三个数字：

```text
run_ns wait_ns slices
```

含义是：

- `run_ns`：这个 task 真正在 CPU 上运行的累计纳秒数。
- `wait_ns`：这个 task 处于可运行状态、但在 runqueue 里等 CPU 的累计纳秒数。
- `slices`：这个 task 被调度上 CPU 的次数。

如果一个线程被 `futex_wake` 后没有立刻运行，而是在 runqueue 里排了一会儿队，`wait_ns` 就会涨。

## 四、实验证明：futex_wake 后，wait_ns 真的会涨

为了让这个现象稳定复现，我们故意制造一个拥挤的 CPU：

- 把主线程、waiter 和 8 个 CPU 忙等线程尽量绑到 CPU 0。
- 主线程先持有 mutex。
- waiter 抢锁失败，睡在 futex 上。
- 8 个 hog 线程一直占 CPU，让运行队列变拥挤。
- 主线程 unlock，触发 futex wake。
- waiter 被唤醒后，要和 hog 线程一起竞争 CPU。
- waiter 真正拿到 CPU 后，立刻读取自己的 `schedstat`。

<a id="code-wakeup-delay"></a>

```c
// 03_wakeup_delay.c
#define _GNU_SOURCE
#include <errno.h>
#include <pthread.h>
#include <sched.h>
#include <stdatomic.h>
#include <stdint.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/syscall.h>
#include <time.h>
#include <unistd.h>

static pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
static pthread_barrier_t barrier;
static atomic_int stop_hogs;
static atomic_long waiter_tid;
static atomic_long unlock_ns;
static unsigned long long run_before_global;
static unsigned long long wait_before_global;
static unsigned long long slices_before_global;

static long gettid_linux(void) {
    return syscall(SYS_gettid);
}

static long now_ns(void) {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return ts.tv_sec * 1000000000L + ts.tv_nsec;
}

static void bind_cpu0(void) {
    cpu_set_t set;
    CPU_ZERO(&set);
    CPU_SET(0, &set);
    int rc = sched_setaffinity(0, sizeof(set), &set);
    if (rc != 0) {
        fprintf(stderr, "sched_setaffinity failed: %s\n", strerror(errno));
    }
}

static void read_schedstat(long tid, unsigned long long *run_ns,
                           unsigned long long *wait_ns,
                           unsigned long long *slices) {
    char path[128];
    snprintf(path, sizeof(path), "/proc/self/task/%ld/schedstat", tid);
    FILE *f = fopen(path, "r");
    if (!f) {
        perror(path);
        exit(1);
    }
    if (fscanf(f, "%llu %llu %llu", run_ns, wait_ns, slices) != 3) {
        fprintf(stderr, "failed to parse %s\n", path);
        exit(1);
    }
    fclose(f);
}

static void *hog(void *arg) {
    bind_cpu0();
    volatile unsigned long x = 0;
    while (!atomic_load(&stop_hogs)) {
        x++;
    }
    return NULL;
}

static void *waiter(void *arg) {
    bind_cpu0();
    atomic_store(&waiter_tid, gettid_linux());
    pthread_barrier_wait(&barrier);

    pthread_mutex_lock(&m);

    long got_ns = now_ns();
    unsigned long long run_ns, wait_ns, slices;
    read_schedstat(gettid_linux(), &run_ns, &wait_ns, &slices);

    printf("waiter got mutex: delay_after_unlock=%ld us\n",
           (got_ns - atomic_load(&unlock_ns)) / 1000);
    printf("waiter schedstat after wake: run_ns=%llu wait_ns=%llu slices=%llu\n",
           run_ns, wait_ns, slices);
    printf("waiter schedstat delta: run_ns=+%llu wait_ns=+%llu slices=+%llu\n",
           run_ns - run_before_global,
           wait_ns - wait_before_global,
           slices - slices_before_global);

    pthread_mutex_unlock(&m);
    return NULL;
}

int main(void) {
    enum { HOGS = 8 };
    pthread_t twait, thog[HOGS];
    unsigned long long run_before, wait_before, slices_before;

    bind_cpu0();
    pthread_barrier_init(&barrier, NULL, 2);
    pthread_mutex_lock(&m);

    pthread_create(&twait, NULL, waiter, NULL);
    pthread_barrier_wait(&barrier);

    for (int i = 0; i < HOGS; i++) {
        pthread_create(&thog[i], NULL, hog, NULL);
    }

    usleep(300000);
    long tid = atomic_load(&waiter_tid);
    read_schedstat(tid, &run_before, &wait_before, &slices_before);
    run_before_global = run_before;
    wait_before_global = wait_before;
    slices_before_global = slices_before;
    printf("waiter schedstat before unlock: run_ns=%llu wait_ns=%llu slices=%llu\n",
           run_before, wait_before, slices_before);

    atomic_store(&unlock_ns, now_ns());
    pthread_mutex_unlock(&m);
    pthread_join(twait, NULL);

    atomic_store(&stop_hogs, 1);
    for (int i = 0; i < HOGS; i++) {
        pthread_join(thog[i], NULL);
    }
    return 0;
}
```

编译运行：

```bash
gcc -O0 -pthread -o 03_wakeup_delay 03_wakeup_delay.c
./03_wakeup_delay
```

真实输出：

```text
waiter schedstat before unlock: run_ns=17792 wait_ns=340708 slices=2
waiter got mutex: delay_after_unlock=2939 us
waiter schedstat after wake: run_ns=17792 wait_ns=3277583 slices=3
waiter schedstat delta: run_ns=+0 wait_ns=+2936875 slices=+1
```

这几个数字很关键：

- `delay_after_unlock=2939 us`：主线程 unlock 后，waiter 不是立刻继续执行，而是过了约 2.9 ms 才真正拿到 CPU、返回到用户态。
- `wait_ns=+2936875`：waiter 的 runqueue 等待时间增加了约 2.94 ms。
- `slices=+1`：waiter 被调度上 CPU 的次数增加了一次。
- `run_ns=+0`：这个采样点非常靠前，waiter 一被调度回来就读取 `schedstat`，这段执行本身还没累计出明显的运行时间。

这就是"唤醒不等于运行"的直接证据。

`pthread_mutex_unlock` 触发 `futex_wake` 后，waiter 先从 futex 等待队列出来，变成可运行状态，再进入 runqueue。它真正继续执行，要等调度器选中它。

## 五、调度器在中间做了什么

普通 pthread 默认走 Linux 的普通公平调度类，也就是 `SCHED_OTHER` / `SCHED_NORMAL` 这条路径。老文章常把这条路径称为 CFS；Linux 6.6 之后主线普通公平调度已经切到 EEVDF。名字变了，但有几个事实对理解 mutex 慢路径最关键：

1. Linux 调度的是 task。
2. 普通线程不是谁刚被唤醒谁就一定马上跑。
3. 被唤醒的线程要进入 CPU 的运行队列。
4. 调度器会在可运行 task 之间做选择。
5. 选择不是简单 FIFO，而是受调度类、nice 权重、公平性、CPU 负载影响。

所以 mutex 慢路径里，`futex_wake` 只负责把等待者推进"可以竞争 CPU"的状态。后面是不是立刻跑，取决于调度器。

先把调度器要看的几层东西拆开。

### 5.1 调度类：普通线程先落在 fair class

Linux 不是只有一种调度策略。常见的有：

- `SCHED_OTHER` / `SCHED_NORMAL`：普通线程默认策略，走公平调度。
- `SCHED_BATCH`：批处理任务，交互性要求低。
- `SCHED_IDLE`：非常低优先级，只在系统很空时跑。
- `SCHED_FIFO` / `SCHED_RR`：实时调度策略，优先级语义完全不同。

我们这里讨论的是默认 pthread，所以重点是 `SCHED_OTHER`。它的 `sched_priority` 是 0；普通线程之间不靠实时优先级抢 CPU，而是靠 nice 值转成的权重，以及公平调度器维护的运行历史来分配 CPU。

这也是为什么上一篇的 mutex 实验里，waiter 被唤醒后不会因为"我是刚被 futex_wake 叫醒的"就天然插队。它只是重新进入普通公平调度系统。

### 5.2 每个 CPU 有自己的 runqueue

Linux 不是把所有 runnable 线程丢进一个全局大队列。每个 CPU 都有自己的运行队列。唤醒一个线程时，内核要决定把它放到哪个 CPU 的 runqueue 上；之后负载均衡逻辑还可能把 task 迁移到别的 CPU。

这件事对 mutex 很重要：

```text
futex_wake
→ waiter 变成 TASK_RUNNING
→ 选择一个 CPU
→ 入队到这个 CPU 的 runqueue
→ 等这个 CPU 的调度器挑它
```

如果这个 CPU 上已经有很多 runnable 线程，waiter 就会排队，`schedstat` 的 `wait_ns` 就会涨。前面的实验故意把 waiter 和 8 个 hog 线程绑到 CPU 0，就是为了让这个排队过程稳定出现。

### 5.3 nice 不是摆设，它会变成调度权重

普通线程没有实时优先级，但有 nice 值。nice 越大，线程越"客气"，权重越低，分到的 CPU 时间越少。

用代码直接量一下。两个线程都绑到 CPU 0，跑满 2 秒：

- `thread0`：nice = 0
- `thread1`：nice = 10

二者都是普通 `SCHED_OTHER`，实时优先级都是 0。唯一差异是 nice。

```c
// 04_scheduler_weight.c
#define _GNU_SOURCE
#include <errno.h>
#include <pthread.h>
#include <sched.h>
#include <stdatomic.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/resource.h>
#include <sys/syscall.h>
#include <time.h>
#include <unistd.h>

struct worker_stat {
    int nice_value;
    long tid;
    int policy;
    int sched_priority;
    unsigned long long run_before;
    unsigned long long wait_before;
    unsigned long long slices_before;
    unsigned long long run_after;
    unsigned long long wait_after;
    unsigned long long slices_after;
    unsigned long long loops;
};

static pthread_barrier_t barrier;
static atomic_int stop_flag;
static struct worker_stat stats[2];

static long gettid_linux(void) {
    return syscall(SYS_gettid);
}

static void bind_cpu0(void) {
    cpu_set_t set;
    CPU_ZERO(&set);
    CPU_SET(0, &set);
    if (sched_setaffinity(0, sizeof(set), &set) != 0) {
        fprintf(stderr, "sched_setaffinity failed: %s\n", strerror(errno));
    }
}

static void read_schedstat(long tid, unsigned long long *run_ns,
                           unsigned long long *wait_ns,
                           unsigned long long *slices) {
    char path[128];
    snprintf(path, sizeof(path), "/proc/self/task/%ld/schedstat", tid);
    FILE *f = fopen(path, "r");
    if (!f) {
        perror(path);
        exit(1);
    }
    if (fscanf(f, "%llu %llu %llu", run_ns, wait_ns, slices) != 3) {
        fprintf(stderr, "failed to parse %s\n", path);
        exit(1);
    }
    fclose(f);
}

static const char *policy_name(int policy) {
    switch (policy) {
    case SCHED_OTHER: return "SCHED_OTHER";
    case SCHED_BATCH: return "SCHED_BATCH";
    case SCHED_IDLE: return "SCHED_IDLE";
    case SCHED_FIFO: return "SCHED_FIFO";
    case SCHED_RR: return "SCHED_RR";
    default: return "UNKNOWN";
    }
}

static void *worker(void *arg) {
    int idx = (int)(long)arg;
    struct worker_stat *s = &stats[idx];
    struct sched_param param;

    bind_cpu0();
    s->tid = gettid_linux();
    if (setpriority(PRIO_PROCESS, s->tid, s->nice_value) != 0) {
        fprintf(stderr, "setpriority tid=%ld nice=%d failed: %s\n",
                s->tid, s->nice_value, strerror(errno));
    }
    s->nice_value = getpriority(PRIO_PROCESS, s->tid);
    pthread_getschedparam(pthread_self(), &s->policy, &param);
    s->sched_priority = param.sched_priority;

    read_schedstat(s->tid, &s->run_before, &s->wait_before, &s->slices_before);
    pthread_barrier_wait(&barrier);

    unsigned long long loops = 0;
    while (!atomic_load(&stop_flag)) {
        loops++;
    }
    s->loops = loops;
    read_schedstat(s->tid, &s->run_after, &s->wait_after, &s->slices_after);
    return NULL;
}

int main(void) {
    pthread_t th[2];
    stats[0].nice_value = 0;
    stats[1].nice_value = 10;

    bind_cpu0();
    pthread_barrier_init(&barrier, NULL, 3);
    pthread_create(&th[0], NULL, worker, (void *)0);
    pthread_create(&th[1], NULL, worker, (void *)1);

    pthread_barrier_wait(&barrier);
    sleep(2);
    atomic_store(&stop_flag, 1);

    pthread_join(th[0], NULL);
    pthread_join(th[1], NULL);

    for (int i = 0; i < 2; i++) {
        struct worker_stat *s = &stats[i];
        printf("thread%d tid=%ld policy=%s sched_priority=%d nice=%d\n",
               i, s->tid, policy_name(s->policy), s->sched_priority, s->nice_value);
        printf("thread%d delta: run_ns=%llu wait_ns=%llu slices=%llu loops=%llu\n",
               i,
               s->run_after - s->run_before,
               s->wait_after - s->wait_before,
               s->slices_after - s->slices_before,
               s->loops);
    }
    return 0;
}
```

编译运行：

```bash
gcc -O2 -pthread -o 04_scheduler_weight 04_scheduler_weight.c
./04_scheduler_weight
```

真实输出：

```text
thread0 tid=14 policy=SCHED_OTHER sched_priority=0 nice=0
thread0 delta: run_ns=1805960830 wait_ns=199058247 slices=82 loops=7713575210
thread1 tid=15 policy=SCHED_OTHER sched_priority=0 nice=10
thread1 delta: run_ns=198891037 wait_ns=1806265206 slices=71 loops=837350484
```

两个线程的调度策略完全一样：

```text
policy = SCHED_OTHER
sched_priority = 0
```

但结果差很多：

- nice=0 的 `thread0` 跑了约 `1.806s` CPU。
- nice=10 的 `thread1` 只跑了约 `0.199s` CPU。
- `thread1` 的 `wait_ns` 反而涨到约 `1.806s`，说明它多数时间都是 runnable，但在 runqueue 里等。

这就证明了：普通调度器不是简单轮流，也不是按唤醒顺序 FIFO。它会根据 task 的权重和运行历史决定谁该拿 CPU。

### 5.4 CFS / EEVDF 到底在公平什么

旧 CFS 的核心直觉是：维护每个可运行任务的虚拟运行时间，谁"欠 CPU"更多，谁更该被选中。nice 权重会影响虚拟时间推进速度：权重高的任务，同样运行一段真实时间，虚拟时间涨得慢，于是更容易继续被选中。

Linux 6.6 之后普通公平调度逐步转到 EEVDF。根据 [Linux 内核 EEVDF 文档](https://docs.kernel.org/scheduler/sched-eevdf.html)，EEVDF 仍然要在同优先级 runnable task 之间分配公平 CPU 时间，只是选择方式变成：计算 task 的 lag 和 virtual deadline，优先选择 eligible 且 virtual deadline 最早的 task。

对这篇文章的主线来说，不需要把公式展开。抓住三点就够：

```text
1. 普通 pthread 默认进入 fair scheduler。
2. fair scheduler 维护每个 task 的运行历史和权重。
3. 被 futex_wake 唤醒的 task 只是加入这套竞争，不是绕过这套竞争。
```

这也解释了为什么上一节的 waiter 被唤醒后，`wait_ns` 还能继续涨：它已经 runnable 了，但调度器还没选它。

### 5.5 唤醒可能触发抢占，但不保证抢占

再往细一点说，`futex_wake` 最终会走到内核的唤醒路径，把 waiter 标成 runnable，并尝试判断它是否值得尽快运行。

如果新唤醒的 task 比当前正在跑的 task 更应该运行，内核可能设置抢占标记，让当前 task 在合适的抢占点让出 CPU。这里有两个限制：

- 它不等于立刻在 `futex_wake` 那一行强行切过去。
- 如果当前 CPU 上还有更合适的 task，或者内核抢占时机还没到，waiter 还是要等。

所以更准确的说法是：

```text
futex_wake 让 waiter 有资格被调度；
调度器决定 waiter 什么时候真正运行。
```

把 mutex unlock 后的路径重新写细一点：

```text
pthread_mutex_unlock
→ futex(FUTEX_WAKE)
→ 内核找到 futex 等待队列
→ 唤醒其中一个 waiter
→ waiter 状态变成 TASK_RUNNING
→ 选择目标 CPU
→ waiter 入队到目标 CPU 的 runqueue
→ fair scheduler 根据权重、运行历史、deadline/lag 等信息选择下一个 task
→ 如果需要，触发抢占
→ context switch
→ waiter 回到用户态，从 futex_wait 返回
→ 重新尝试拿 mutex
```

画成分层图更直观：

```text
时间向下

线程 A：持锁者 / unlock 路径                    线程 B：waiter / 被唤醒路径
──────────────────────────────────────────────────────────────────────────────
用户态
  pthread_mutex_unlock(&m)
        │
        │ 发现 __lock 表示可能有人等待
        ▼
  futex(FUTEX_WAKE, &m.__lock, 1)
        │
        │ system call
        ▼
──────────────────────────────────────────────────────────────────────────────
内核：futex 子系统
  根据 futex word 地址计算 key
        │
        ▼
  找到 futex hash bucket
        │
        ▼
  找到挂在这个 key 上的 waiter task  ───────────────► 线程 B 仍在睡眠
        │                                           State: S
        │                                           wchan: futex_wait_queue
        ▼
  futex_wake_mark()
        │
        ▼
──────────────────────────────────────────────────────────────────────────────
内核：唤醒路径
  try_to_wake_up(task B)
        │
        ├─ 把 B 从等待队列摘下来
        │
        ├─ B.state: S → TASK_RUNNING
        │
        ├─ 选择目标 CPU
        │
        └─ enqueue_task(B, target_rq)
                         │
                         ▼
──────────────────────────────────────────────────────────────────────────────
内核：目标 CPU 的 runqueue
  rq(CPU x)
    ├─ 当前正在跑的 task
    ├─ 其他 runnable task
    ├─ 线程 B  ← 刚被唤醒，只是进入队列
    └─ ...
        │
        │ B 在这里等待时：
        │   /proc/<pid>/task/<tid>/schedstat 的 wait_ns 增加
        ▼
──────────────────────────────────────────────────────────────────────────────
内核：fair scheduler
  pick_next_task()
        │
        ├─ 看调度类：SCHED_OTHER / fair class
        ├─ 看权重：nice → weight
        ├─ 看运行历史：谁已经跑得多，谁还欠 CPU
        ├─ EEVDF 下还会看 lag / virtual deadline
        └─ 选中线程 B
             │
             ▼
──────────────────────────────────────────────────────────────────────────────
内核：上下文切换
  context_switch(prev, B)
        │
        ├─ 保存 prev 的 CPU 上下文
        ├─ 切换当前 task 指针
        ├─ 切换内核栈 / 寄存器状态
        └─ 恢复 B 的执行现场
             │
             ▼
──────────────────────────────────────────────────────────────────────────────
线程 B：重新回到用户态
  futex(FUTEX_WAIT) 返回
        │
        ▼
  pthread_mutex_lock 慢路径继续
        │
        ▼
  重新检查 / 重新 CAS 抢锁
        │
        ├─ 抢到：进入临界区
        └─ 没抢到：可能再次 futex_wait
```

这张图里最关键的是中间两段：

```text
S → TASK_RUNNING
```

只是"睡眠变成可运行"；

```text
runqueue → pick_next_task → context_switch
```

才是"真正重新获得 CPU"。

注意最后一步：被唤醒后并不是自动拥有 mutex。它只是有资格继续执行，回到用户态后还要重新走 lock 逻辑。

## 六、上下文切换：慢路径贵在哪里

上一篇已经证明：无竞争的 `pthread_mutex_lock` 可以完全不进内核；有竞争时，会陷入 futex 等待。

调度角度再看一次：

```text
无竞争：
CAS 成功
不睡眠
不唤醒
不进入 runqueue 排队
没有因为锁产生上下文切换

有竞争：
CAS 失败
futex_wait 睡眠
当前线程让出 CPU
unlock 时 futex_wake
线程进入 runqueue
等待调度
上下文切换回来
重新抢锁
```

这个差异可以用 `getrusage(RUSAGE_SELF)` 直接数出来。这里统计两个字段：

- `ru_nvcsw`：自愿上下文切换。比如线程主动睡眠、等锁、等 I/O。
- `ru_nivcsw`：非自愿上下文切换。比如时间片到了，被调度器强制换下。

<a id="code-switches"></a>

```c
// 04_switches.c
#define _GNU_SOURCE
#include <pthread.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/resource.h>
#include <time.h>
#include <unistd.h>

static pthread_mutex_t m = PTHREAD_MUTEX_INITIALIZER;
static long rounds;
static long hold_us;
static long shared;

static double now_ms(void) {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return ts.tv_sec * 1000.0 + ts.tv_nsec / 1000000.0;
}

static void *worker(void *arg) {
    for (long i = 0; i < rounds; i++) {
        pthread_mutex_lock(&m);
        shared++;
        if (hold_us > 0) {
            usleep(hold_us);
        }
        pthread_mutex_unlock(&m);
    }
    return NULL;
}

int main(int argc, char **argv) {
    if (argc != 4) {
        fprintf(stderr, "usage: %s <threads> <rounds> <hold_us>\n", argv[0]);
        return 1;
    }

    int threads = atoi(argv[1]);
    rounds = atol(argv[2]);
    hold_us = atol(argv[3]);

    pthread_t *th = calloc(threads, sizeof(*th));
    struct rusage before, after;
    getrusage(RUSAGE_SELF, &before);

    double start = now_ms();
    for (int i = 0; i < threads; i++) {
        pthread_create(&th[i], NULL, worker, NULL);
    }
    for (int i = 0; i < threads; i++) {
        pthread_join(th[i], NULL);
    }
    double end = now_ms();
    getrusage(RUSAGE_SELF, &after);

    printf("threads=%d rounds=%ld hold_us=%ld shared=%ld elapsed=%.1f ms\n",
           threads, rounds, hold_us, shared, end - start);
    printf("voluntary_context_switches=%ld\n",
           after.ru_nvcsw - before.ru_nvcsw);
    printf("involuntary_context_switches=%ld\n",
           after.ru_nivcsw - before.ru_nivcsw);
    return 0;
}
```

编译：

```bash
gcc -O2 -pthread -o 04_switches 04_switches.c
```

先跑单线程、无竞争、临界区不睡眠：

```bash
./04_switches 1 1000000 0
```

真实输出：

```text
threads=1 rounds=1000000 hold_us=0 shared=1000000 elapsed=4.6 ms
voluntary_context_switches=1
involuntary_context_switches=0
```

100 万次 lock/unlock，几乎没有上下文切换。那个 `1` 基本来自线程创建/退出/join 这类外围动作，不是 mutex 竞争本身。

再跑 4 个线程竞争同一把锁，并且每次持锁时睡 1 ms：

```bash
./04_switches 4 100 1000
```

真实输出：

```text
threads=4 rounds=100 hold_us=1000 shared=400 elapsed=800.5 ms
voluntary_context_switches=703
involuntary_context_switches=0
```

总共只有 400 次临界区进入，但自愿上下文切换涨到了 703 次。

为什么是自愿切换？因为抢不到锁的线程不是被时间片强行换下，而是自己进了 `futex_wait`，主动睡眠、让出 CPU。

这也解释了为什么锁竞争会慢：

```text
不是 pthread_mutex_lock 这个函数天然慢；
是竞争把它从用户态 CAS 路径，推到了内核睡眠/唤醒/调度/上下文切换路径。
```

## 七、把整条链路串起来

现在可以把上一篇和这一篇合在一起看。

无竞争时：

```text
线程 A
  │
  ├─ pthread_mutex_lock
  │    └─ CAS: __lock 0 → 1 成功
  │
  ├─ 临界区
  │
  └─ pthread_mutex_unlock
       └─ __lock 1 → 0

全程用户态
不进 futex
不睡眠
不调度
```

有竞争时：

```text
线程 A                                      线程 B
  │                                           │
  ├─ pthread_mutex_lock                       │
  │    └─ CAS 成功，持锁                     │
  │                                           ├─ pthread_mutex_lock
  │                                           │    └─ CAS 失败
  │                                           │    └─ futex(FUTEX_WAIT)
  │                                           │    └─ State: S
  │                                           │    └─ wchan: futex_wait_queue
  │                                           │
  ├─ pthread_mutex_unlock                     │
  │    └─ futex(FUTEX_WAKE)                  │
  │                                           │
  │                                           ├─ TASK_RUNNING
  │                                           ├─ 进入 runqueue
  │                                           ├─ wait_ns 增加
  │                                           ├─ 被调度器选中
  │                                           └─ 返回用户态，继续抢锁
```

这条链路里，`futex_wake` 只完成了一半：

```text
睡眠 → 可运行
```

另一半要交给调度器：

```text
可运行 → 正在运行
```

这就是为什么"被唤醒"不等于"立刻运行"。

## 八、回到开头的问题

现在回答开头的问题：线程被 `futex_wake` 唤醒后，到底发生了什么？

**第一，它从 futex 等待队列里出来。**

上一篇看到的是 `futex_wait`，这一篇用 `wchan=futex_wait_queue` 坐实了睡眠位置。

**第二，它从 sleeping 变成 runnable。**

也就是进入 `TASK_RUNNING`。但这个名字不是"正在运行"，而是"可以运行"。

**第三，它进入某个 CPU 的 runqueue。**

这时它还没有真正执行用户态代码。实验里 `wait_ns=+2936875`，说明它已经可运行，但还在队列里等了约 2.94 ms。

**第四，调度器选中它，发生上下文切换。**

它重新获得 CPU，从内核返回用户态，`pthread_mutex_lock` 的慢路径继续往下走。

**第五，它还要重新抢锁。**

futex wake 不是把锁直接交给它。wake 只是告诉它："条件可能变了，你可以醒来再试一次。"

所以 mutex 竞争真正贵的地方，不只是一次系统调用，而是整条路径：

```text
用户态 CAS 失败
→ futex_wait
→ 线程睡眠
→ 上下文切换出去
→ futex_wake
→ 进入 runqueue
→ 调度等待
→ 上下文切换回来
→ 重新抢锁
```

这就是上一篇里"同一个 `pthread_mutex_lock`，快的时候几纳秒，慢的时候微秒到毫秒"背后的操作系统机制。

下一步如果还要继续挖，可以沿两条线走：

- 往 glibc 挖：不同类型的 mutex，尤其是 adaptive、robust、PI mutex。
- 往内核挖：`try_to_wake_up()`、runqueue、普通调度类，以及 CFS/EEVDF 怎么决定下一个 task。

但到这里，解释普通 mutex 慢路径已经够用了：**锁竞争不是简单地"等一下"，而是把线程送进了 Linux 的睡眠、唤醒、排队和调度系统。**
